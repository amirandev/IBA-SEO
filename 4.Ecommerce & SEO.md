## **სლაიდი 3: HTML ბმულები & SEO**

> თეორია: ბმულები არის ინტერნეტის საფუძველი. Google-ის PageRank ალგორითმი აფასებს გვერდის ავტორიტეტს მასზე მიმთითებელი ბმულების რაოდენობითა და ხარისხით. სემანტიკური ბმულები ეხმარება საძიებო სისტემებს კონტექსტის გაგებაში.

### **რატომ არის ბმულები მნიშვნელოვანი:**

ბმულები არის საფუძველი:
- **ნავიგაცია** (მომხმარებლის გამოცდილება)
- **დათვალიერებადობა** (საძიებო სისტემები პოულობენ თქვენს გვერდებს)
- **გვერდის ავტორიტეტი** (ლინკ წვენის განაწილება)

### **საუკეთესო პრაქტიკა ბმულებისთვის:**

```html
<!-- ✅ GOOD: Descriptive, semantic links -->
<a href="/products/wireless-bluetooth-headphones" 
   title="Premium Wireless Bluetooth Headphones with Noise Cancelling">
   Wireless Bluetooth Headphones
</a>

<!-- ❌ BAD: Generic, non-descriptive -->
<a href="/product.php?id=123">Click here</a>
```

### **ძირითადი ბმულის ატრიბუტები:**

1. **`href`** - დანიშნულების URL (გამოიყენეთ სუფთა, წაკითხვადი მისამართები)
2. **`title`** - ტულტიპის ტექსტი (აწვდის კონტექსტს)
3. **`rel`** - ურთიერთობა (nofollow, noopener, და ა.შ.)
4. **`target`** - სად გაიხსნას (_blank, _self)

## **სლაიდი 4: Title ატრიბუტის დეტალური მიმოხილვა**

> თეორია: title ატრიბუტი წარმოადგენს დამატებით სემანტიკურ ფენას. მიუხედავად იმისა, რომ Google-მა შეამცირა მისი მნიშვნელობა რანჟირებაში, ის მაინც მნიშვნელოვანია ხელმისაწვდომობისა და მომხმარებლის გამოცდილებისთვის.

### **რა არის Title ატრიბუტი?**

`title` ატრიბუტი გვაწვდის **დამატებით ინფორმაციას** ელემენტის შესახებ, ნაჩვენებია ტულტიპის სახით ჰოვერის დროს.

### **გავლენა SEO-ზე:**
- ✅ აუმჯობესებს **ხელმისაწვდომობას** 
- ✅ აწვდის **კონტექსტს** მომხმარებლებს
- ✅ ეხმარება საძიებო სისტემებს **ბმულის რელევანტურობის** გაგებაში
- ⚠️ გამოიყენეთ ზომიერად - ნუ გადატვირთავთ საკვანძო სიტყვებით!

### **მაგალითები:**

```html

   Steel Pipe Fitting
```

### **საუკეთესო პრაქტიკა:**
- შეინახეთ **მოკლედ** (50-60 სიმბოლო)

## **სლაიდი 5: Alt ატრიბუტი სურათებისთვის**

> თეორია: საძიებო სისტემები ვერ "ხედავენ" სურათებს — ისინი კითხულობენ alt ტექსტს. alt ტექსტი უზრუნველყოფს ხელმისაწვდომობას და ეხმარება Google-ს სურათების ინდექსირებაში, რაც მნიშვნელოვანია ვიზუალური ძიებისთვის.

### **რატომ არის Alt ტექსტი კრიტიკული ელექტრონული კომერციისთვის:**

1. **ხელმისაწვდომობა** - ეკრანის წამკითხველები აღწერენ სურათებს მხედველობის შეზღუდული მქონე მომხმარებლებისთვის
2. **SEO** - Google-ს არ შეუძლია სურათების "დანახვა", ის კითხულობს alt ტექსტს
3. **სათადარიგო** - გამოჩნდება, როდესაც სურათები არ იტვირთება
4. **სურათების ძიება** - ჩნდება Google-ის სურათების ძიებაში

### **Alt ტექსტის სწორი განხორციელება:**

```html
<!-- ✅ EXCELLENT: Descriptive, includes product details -->
<img src="steel-pipe-fitting.jpg" 
     alt="ფოლადის მილტუჩი MAMU ბრენდი, მოდელი 925872, SKU: 446310786"
     width="500" 
     height="500">

<!-- ✅ GOOD: Clear and descriptive -->
<img src="headphones.jpg" 
     alt="Wireless Bluetooth Headphones - Noise Cancelling, Black">

<!-- ❌ BAD: Too generic -->
<img src="product.jpg" alt="Product">

<!-- ❌ BAD: Keyword stuffing -->
<img src="pipe.jpg" alt="pipe fitting steel pipe best price buy pipe fitting">

<!-- ✅ DECORATIVE: Empty alt (screen readers skip it) -->
<img src="decorative-border.png" alt="">
```

### **ელექტრონული კომერციის სურათების Alt ტექსტის ფორმულა:**

```
[Product Name] + [Brand] + [Key Feature/Color/Size] + [Model/SKU if relevant]
```

**მაგალითი:**
```html

     alt="MacBook Pro 16-inch, Apple M2 Chip, 32GB RAM, Space Gray">
```

### **გავლენა SEO-ზე:**
- სურათები სწორი alt ტექსტით რეიტინგდებიან **უფრო მაღლა სურათების ძიებაში**
- ხელს უწყობს **გვერდის საერთო რელევანტურობას**
- აუმჯობესებს **ხელმისაწვდომობის ქულას** (WCAG-სთან შესაბამისობა)

## **სლაიდი 6: ელექტრონული კომერციის გვერდის ბმულების სტრუქტურა**


## **სლაიდი 7: International & hreflang SEO**

> თეორია: Google-ის მიზანია აჩვენოს მომხმარებლის ენასა და რეგიონთან შესაბამისი შედეგები. hreflang ტეგები არის სიგნალი, რომელიც ეხმარება Google-ს აირჩიოს სწორი ენობრივი ვერსია. მის გარეშე Google თავად წყვეტს, რამაც შეიძლება არასწორ შედეგამდე მიგვიყვანოს.

### **რატომ არის International SEO მნიშვნელოვანი:**

მრავალენოვანი ელექტრონული კომერციის საიტებისთვის, საძიებო სისტემებმა უნდა იცოდნენ, ენის/რეგიონის რომელი ვერსია აჩვენონ შედეგებში. სათანადო სიგნალების გარეშე, მომხმარებლები ხედავენ ენის არასწორ ვერსიას ან Google თარგმანებს დუბლირებულ კონტენტად განიხილავს.

### **hreflang-ის იმპლემენტაცია:**

```html
<!-- ✅ ენის მიზნობრიობა ქართულისა და ინგლისურისთვის -->
<link rel="alternate" hreflang="ka" href="https://ltdmamu.ge/ka/product/steel-pipe-fitting" />
<link rel="alternate" hreflang="en" href="https://ltdmamu.ge/en/product/steel-pipe-fitting" />
<link rel="alternate" hreflang="x-default" href="https://ltdmamu.ge/ka/product/steel-pipe-fitting" />

<!-- ✅ ასევე შეგიძლიათ გამოიყენოთ HTTP სათაურები ან საიტის რუკები -->
Link: <https://ltdmamu.ge/ka/product/steel-pipe-fitting>; rel="alternate"; hreflang="ka",
      <https://ltdmamu.ge/en/product/steel-pipe-fitting>; rel="alternate"; hreflang="en"
```

### **hreflang-ის საუკეთესო პრაქტიკა:**

- ✅ გამოიყენეთ **სრული აბსოლუტური URL-ები** (არა ფარდობითი ბილიკები)
- ✅ ჩართეთ **თვითრეფერენსული** hreflang (გვერდი საკუთარ თავზე მიუთითებს)
- ✅ გამოიყენეთ **x-default** როგორც სათადარიგო (ჩვეულებრივ, ძირითადი ენა)
- ✅ გამოიყენეთ **ISO 639-1** ენის კოდები (`ka`, `en`, `de`)
- ✅ რეგიონის კოდები დაამატეთ მხოლოდ საჭიროების შემთხვევაში: `en-US`, `en-GB`, `ka-GE`
- ⚠️ არასოდეს აწინააღმდეგოთ hreflang-ისა და canonical-ის ტეგები — მაგალითად, თუ hreflang მიუთითებს ინგლისურ გვერდზე, canonical არ უნდა მიუთითებდეს ქართულზე. Google ვერ გადაწყვეტს რომელი ვერსია აჩვენოს და შეიძლება არცერთი არ აჩვენოს.

### **ენის ამომრჩევლის UX:**

```html
<nav class="language-switcher" aria-label="Language selector">
  <a href="/ka/product" hreflang="ka" lang="ka" rel="alternate">ქართული</a>
  <a href="/en/product" hreflang="en" lang="en" rel="alternate">English</a>
</nav>
```

| სტრატეგია | მაგალითი | უპირატესობები | ნაკლოვანებები |

| **ქვედირექტორია** | `ltdmamu.ge/ka/` | მარტივი მართვა, ძლიერი გეო-სიგნალი | URL-ის რესტრუქტურიზაციაა საჭირო |

## **სლაიდი 8: JSON-LD Structured Data**

> თეორია: სტრუქტურირებული მონაცემები (Structured Data) ეფუძნება Schema.org-ის ვოკაბულარს — Google-ს, Bing-სა და Yandex-ის ერთობლივ სტანდარტს. JSON-LD არის Google-ის რეკომენდებული ფორმატი, რადგან ის არ ერევა HTML-ში.

Structured Data (სტრუქტურირებული მონაცემები) არის მონაცემთა ფორმატი, რომელიც ეხმარება საძიებო სისტემებს გაიგონ თქვენი გვერდის შინაარსი და აჩვენონ გამორჩეული შედეგები.

### **რა არის Structured Data?**

Structured data (Schema.org მარკირება) ეხმარება საძიებო სისტემებს **გაიგონ თქვენი კონტენტი** და აჩვენონ **მდიდარი სნიპეტები** ძიების შედეგებში.

### **რატომ JSON-LD?**
- ✅ **Google-ის რეკომენდებული ფორმატი**
- ✅ მარტივი იმპლემენტაცია
- ✅ არ ახდენს გავლენას HTML სტრუქტურაზე
- ✅ განთავსებულია `<head>`-ში ან `</body>`-მდე

## **სლაიდი 17: JavaScript-ის რენდერინგი & SEO**

> თეორია: Googlebot მუშაობს ორეტაპიანად — ჯერ ათვალიერებს HTML-ს, შემდეგ რენდერავს JS-ს. თუ JavaScript არ არის ოპტიმიზირებული, Googlebot შეიძლება ვერ მოასწროს რენდერინგი, რაც გამოიწვევს კონტენტის არაინდექსირებას.

### **როგორ რენდერავს Google JavaScript-ს:**

Googlebot ამუშავებს JavaScript-ს ორეტაპიანი სისტემით:
1. **Crawl** - აღმოაჩენს URL-ებს, რიგში აყენებს გვერდებს რენდერინგისთვის
2. **Render** - ასრულებს JS-ს, ინდექსირებს საბოლოო DOM-ს

**პრობლემა:** თუ Googlebot ვერ ახერხებს თქვენი JS-ის სწორად რენდერინგს, თქვენი კონტენტი არ ინდექსირდება.

### **რენდერინგის სტრატეგიები ელექტრონული კომერციისთვის:**

```html
<!-- ✅ SSR (Server-Side Rendering) - საუკეთესოა SEO-სთვის -->
<!-- სერვერი აგზავნის სრულად გადამუშავებულ HTML-ს -->
<!-- Googlebot ხედავს სრულ კონტენტს დაუყოვნებლივ -->
<div id="product">
  <h1>Steel Pipe Fitting</h1>
  <p class="price">25.99 GEL</p>
  <img src="product.jpg" alt="Steel Pipe Fitting">
</div>

<!-- ⚠️ CSR (Client-Side Rendering) - სარისკოა -->
<!-- სერვერი აგზავნის ცარიელ გარსს, JS რენდერავს კონტენტს -->
<!-- Googlebot შესაძლოა არ დაელოდოს JS-ის შესრულებას -->
<div id="root"></div>
<script src="app.js"></script>

<!-- ✅ SSG (Static Site Generation) - საუკეთესოა შესრულებისთვის -->
<!-- წინასწარ აწყობილი HTML build-ის დროს, სერვერი არ არის საჭირო -->
```

### **Framework-ების რეკომენდაციები:**

| Framework | SEO მიდგომა | ელექტრონული კომერციის ვარგისიანობა |
|---|---|---|
| **Next.js** | SSR + SSG | შესანიშნავი (React) |
| **Nuxt.js** | SSR + SSG | შესანიშნავი (Vue) |
| **Remix** | SSR | ძალიან კარგი |
| **Astro** | SSG + ნაწილობრივი ჰიდრატაცია | შესანიშნავი |
| **SPA (React/Vue CSR)** | საჭიროებს წინასწარ რენდერინგს | სუსტია გამოსავლების გარეშე |

### **კრიტიკული პრაქტიკები:**
```html
<!-- ✅ გამოიყენეთ server-side rendering ან static generation -->
<!-- ✅ დანერგეთ dynamic rendering უკანასკნელ საშუალებად -->

<!-- ✅ დარწმუნდით, რომ კრიტიკული კონტენტი საწყის HTML-შია -->
<title>Steel Pipe Fitting | LTD MAMU</title>
<meta name="description" content="High-quality steel pipe fitting...">

<!-- ✅ გამოიყენეთ სათანადო სემანტიკური HTML (არა მხოლოდ div-ები) -->
<nav aria-label="Breadcrumb">...</nav>
<main>
  <article>
    <h1>Product Name</h1>
  </article>
</main>
```

## **სლაიდი 18: Core Web Vitals (ძირითადი ვებ-ვიტალები)**

> თეორია: Core Web Vitals ეფუძნება რეალური მომხმარებლის გამოცდილების გაზომვას (RUM). Google იყენებს Chrome-ის მონაცემებს (CrUX) რეალური მომხმარებლების ჩატვირთვის, ინტერაქტიულობისა და ვიზუალური სტაბილურობის შესაფასებლად.

### **რა არის Core Web Vitals?**

Google-ის მომხმარებელზე ორიენტირებული შესრულების მეტრიკები, რომლებიც გავლენას ახდენენ საძიებო რეიტინგზე:

| მეტრიკა | რას ზომავს | კარგი ქულა | გავლენა ელექტრონულ კომერციაზე |
|---|---|---|---|
| **LCP** (Largest Contentful Paint) | ჩატვირთვის სიჩქარე | < 2.5s | პროდუქტის სურათები, hero ბანერები |
| **INP** (Interaction to Next Paint) | ინტერაქტიულობა | < 200ms | კალათაში დამატება, ფილტრის დაჭერა |
| **CLS** (Cumulative Layout Shift) | ვიზუალური სტაბილურობა | < 0.1 | სურათები ზომების გარეშე, რეკლამები |

### **LCP ოპტიმიზაცია ელექტრონული კომერციისთვის:**

```html
<!-- ✅ ოპტიმიზაცია გაუკეთეთ hero სურათებს - LCP-ის ყველაზე დიდი წყარო -->
<img src="product-hero.webp" 
     alt="Steel Pipe Fitting" 
     fetchpriority="high"
     width="800" 
     height="800">

<!-- ✅ წინასწარ ჩატვირთეთ LCP სურათი -->
<link rel="preload" href="/images/product-hero.webp" as="image">

<!-- ✅ გამოიყენეთ თანამედროვე ფორმატები -->
<picture>
  <source srcset="product.avif" type="image/avif">
  <source srcset="product.webp" type="image/webp">
  <img src="product.jpg" alt="Product" width="800" height="800">
</picture>

<!-- ✅ გამოიყენეთ CDN სურათების ოპტიმიზაციით -->
https://cdn.ltdmamu.ge/fit=cover,width=800/product.webp
```

### **INP ოპტიმიზაცია:**

```html

<script src="vendor+app+checkout.js"></script>

<!-- ✅ კარგი: დაყოფილი მარშრუტის მიხედვით -->

<script src="product-page.js"></script>

<!-- ✅ მოერიდეთ ხანგრძლივ tasks-ებს (>50ms) -->
<script>
  // ❌ ბლოკავს მთავარ თრედს
  heavyComputation();

  // ✅ ყოფს სამუშაოს მიკროტასკებად
```

## **სლაიდი 19: ელექტრონული კომერციის სპეციფიკური SEO**

> თეორია: ელექტრონული კომერციის SEO განსხვავდება ჩვეულებრივი SEO-სგან, რადგან აქ გვაქვს ათასობით პროდუქტის გვერდი, კატეგორიები, ფილტრები და მუდმივად ცვალებადი კონტენტი. Google-ის ალგორითმი აფასებს თითოეულ გვერდს ინდივიდუალურად.

### **პროდუქტის გვერდის საკონტროლო სია:**

✅ **უნიკალური, აღწერილობითი სათაური** (50-60 სიმბოლო)  
✅ **მიმზიდველი meta აღწერა** (155-160 სიმბოლო)  
✅ **მაღალი ხარისხის სურათები** alt ტექსტით  
✅ **გასაგები ფასები** და ხელმისაწვდომობა  
✅ **მომხმარებლის მიმოხილვები** schema-თ  
✅ **Breadcrumb ნავიგაცია**  
✅ **მსგავსი პროდუქტები** (შიდა ლინკინგი)  
✅ **Mobile-ზე რესპონსიული** დიზაინი  
✅ **სწრაფი ჩატვირთვის** სიჩქარე (<3 წამი)  
✅ **SSL სერტიფიკატი** (HTTPS)  
✅ **სტრუქტურირებული მონაცემები** (Product, Offer, Review) 

### **კატეგორიის გვერდის SEO:**

```html
<!-- კატეგორიის გვერდი აღწერით -->
```

## **სლაიდი 21: ფასეტური (Faceted) ნავიგაციის SEO**

> თეორია: ფასეტური ნავიგაცია ქმნის ინდექსის გაბერვის (index bloat) პრობლემას. Google-ის crawl budget თითოეული საიტისთვის შეზღუდულია — თუ Google ხარჯავს მას ფილტრის გვერდებზე, ნაკლები რჩება მნიშვნელოვანი გვერდებისთვის.

### **Faceted ნავიგაციის პრობლემა:**

ფილტრები, დალაგების ვარიანტები და პარამეტრებზე დაფუძნებული ნავიგაცია შექმნის ათასობით მსგავს URL-ს:
```
/products?color=steel&size=12mm&sort=price
/products?color=steel&size=15mm&sort=price  
/products?color=brass&size=12mm&sort=newest
<!-- ეს მრავლდება ექსპონენციალურად! -->
```

### **Facet-ების მართვის გადაწყვეტილებები:**

```html
<!-- ✅ 1. noindex ფილტრის გვერდები -->
<meta name="robots" content="noindex, follow">

<!-- ✅ 2. გამოიყენეთ AJAX/History API ფილტრებისთვის (URL-ის ცვლილების გარეშე) -->
<script>
  // განაახლეთ URL hash query params-ის ნაცვლად
  window.history.replaceState({}, '', '#filter=steel');
</script>

<!-- ✅ 3. Canonical მშობელ კატეგორიაზე -->
<link rel="canonical" href="https://ltdmamu.ge/pipe-fittings">

<!-- ✅ 4. დაუშვით კონკრეტული სასარგებლო facet URL-ები, დაბლოკეთ დანარჩენი -->
<!-- დატოვეთ: /pipe-fittings?material=steel (სასარგებლო მომხმარებლებისთვის) -->
<!-- დაბლოკეთ: /pipe-fittings?material=steel&sort=price&page=2 (კომბინაციური) -->
```

### **Facet-ის დამუშავების გადაწყვეტილების მატრიცა:**

| Facet-ის ტიპი | SEO სტრატეგია | მაგალითი |
|---|---|---|
| **მაღალი ღირებულების** (მასალა, ზომა) | ინდექსირება canonical-ით | `/pipe-fittings?material=steel` |
| **დაბალი ღირებულების** (ფერი, დალაგება) | `noindex` ან AJAX | `/pipe-fittings?sort=price` |
| **კომბინაციური** (მრავალარჩევიანი) | `noindex` | `/pipe-fittings?material=steel&color=silver` |
| **ძიების შედეგები** | `noindex` | `/search?q=steel+pipe` |

### **ძირითადი პრინციპები:**
1. **Crawl budget** - Google crawls-ავს შეზღუდული რაოდენობის გვერდებს თითოეული საიტიდან
2. **თხელი კონტენტი** - ფილტრის გვერდების უმეტესობას აქვს მინიმალური უნიკალური კონტენტი
3. **დუბლიკატი კონტენტი** - მრავალჯერადი ფილტრის კომბინაციები იწვევს ერთსა და იმავე შედეგებს
4. **გამოიყენეთ robots.txt** პარამეტრებით დატვირთული URL-ების დასაბლოკად
```
User-agent: *
Disallow: /*?*sort=
Disallow: /*?*page=
Allow: /pipe-fittings?material=
```

## **სლაიდი 22: პაგინაცია და თხელი კონტენტი**

> თეორია: Google-ის ალგორითმი ანიჭებს რეიტინგის ძირითად წილს გვერდის პირველ ვერსიას. ღრმა პაგინირებული გვერდები იღებენ მინიმალურ ავტორიტეტს. ამიტომ მნიშვნელოვანია, რომ მომხმარებელი სწრაფად მივიდეს სასურველ პროდუქტამდე.

### **პაგინაციის SEO გამოწვევები:**

პროდუქტების ჩამონათვალის გვერდები (PLPs) ასობით ელემენტით ქმნის პაგინაციის პრობლემებს:
- გვერდი 1 იღებს მთელ რეიტინგის ძალას
- ღრმა გვერდებს აქვთ თხელი კონტენტი
- Infinite scroll ართულებს crawling-ს

### **პაგინაციის იმპლემენტაცია:**

```html
<!-- ✅ rel=next/prev (Google ახლა განიხილავს მათ მინიშნებებად) -->

<link rel="next" href="/pipe-fittings?page=4">

<!-- View-All გვერდი აჩვენებს ყველა პროდუქტს, პაგინირებული გვერდები canonical-ით მიუთითებენ მასზე -->
```

### **თხელი კონტენტის პრევენცია:**

```html
<!-- კატეგორიის გვერდი მდიდარი აღწერით გვერდზე 1 -->

<div class="product-listing">
  <!-- პროდუქტების ბადე (პაგინირებული) -->

```

### **საუკეთესო პრაქტიკა:**
1. **მდიდარი კატეგორიების აღწერილობები** - დაამატეთ უნიკალური კონტენტი პროდუქტების ჩამონათვალის ზემოთ
2. **View-All ვარიანტი** - მიეცით მომხმარებლებს საშუალება ნახონ ყველა პროდუქტი ერთ გვერდზე (canonical-ით)
3. **`noindex` ღრმა გვერდები** - განიხილეთ `noindex, follow` მე-3 გვერდიდან+
4. **Infinite scroll + ისტორია** - განაახლეთ URL და pushState ღრმა ბმულების გვერდებისთვის
5. **Load more ღილაკი** > Infinite scroll (crawlable)
6. **სტრუქტურირებული მონაცემები თითოეული გვერდისთვის** - თითოეულ პაგინირებულ გვერდს უნდა ჰქონდეს საკუთარი breadcrumb

### **პროდუქტების რაოდენობის სტრატეგია:**

## **სლაიდი 24: 404-ები, Redirect-ები და მარაგის ამოწურვის მართვა**

| შეწყვეტილი პროდუქტი | **410 Gone** | ინდექსიდან სწრაფი მოცილება |
| მუდმივად გადატანილი | **301 Redirect** | გადააქვს ლინკის ექვითი |
| დროებით გადატანილი | **302 Redirect** | ინახავს ძველი გვერდის ინდექსირებას |
| წაშლილი / არ მოიძებნა | **404 Not Found** | ბუნებრივად იშლება |

### **Redirect-ის საუკეთესო პრაქტიკა:**
```html
<!-- ✅ Server-side 301 redirect (htaccess / nginx) -->
Redirect 301 /old-product-page /new-product-page

<!-- ❌ მოერიდეთ: Client-side redirect-ებს (JS ან meta refresh) -->
<meta http-equiv="refresh" content="0; url=/new-page">
```

### **Soft 404-ები, რომლებსაც თავი უნდა აარიდოთ:**
```html
<!-- ❌ ცუდი: მარაგის ამოწურვის გვერდი აბრუნებს 200-ს, მაგრამ აჩვენებს "პროდუქტები არ არის" -->
<!--    Google ხედავს ამას როგორც soft 404-ს -->

<!-- ✅ კარგი: დააბრუნეთ 200 დროებით მარაგის ამოწურვისთვის, აჩვენეთ მსგავსი ალტერნატივები -->
<section class="alternatives">
  <h2>მსგავსი პროდუქტები, რომლებიც შეიძლება მოგეწონოთ</h2>
  <a href="/pipe-fitting-brass">Brass Pipe Fitting</a>
</section>
```

### **ძირითადი პრინციპები:**
1. **შეინახეთ OOS გვერდები აქტიური** - ნუ წაშლით, აჩვენეთ ალტერნატივები
2. **გამოიყენეთ 301 redirect-ები** მუდმივი გადატანებისთვის, არასდროს 302
3. **410 შეწყვეტილისთვის** - უფრო სწრაფი ინდექსიდან მოცილება ვიდრე 404
4. **მოერიდეთ soft 404-ებს** - თითოეული არის დაკარგული რეიტინგის შესაძლებლობა
5. **მონიტორინგი Search Console-ში** - Crawl errors განყოფილება

## **სლაიდი 26: სწრაფი მითითების ჩამონათვალი**

> თეორია: ეს სლაიდი არის სწრაფი მითითება ყველაზე მნიშვნელოვანი HTML ატრიბუტებისა და Schema თვისებების შესახებ, რომლებიც პირდაპირ გავლენას ახდენენ SEO-ზე.

### **აუცილებელი HTML ატრიბუტები:**
```html
<!-- ლინკები -->
<a href="/product" 
   title="Descriptive tooltip"
   rel="noopener noreferrer"
   target="_blank">Link Text</a>

<!-- სურათები -->
<img src="image.webp" 
     alt="Descriptive alt text"
     width="500"
     height="500"
     loading="lazy">

<!-- Meta ტეგები -->
<meta name="description" content="155-160 character description">
<meta name="robots" content="index, follow">
<link rel="canonical" href="https://example.com/preferred-url">
```

### **საჭირო Schema თვისებები:**

**Product:**
- name ✅
- image ✅
- offers (price, priceCurrency, availability) ✅

**Organization:**
- name ✅
- url ✅
- logo ✅

**Review:**
- author ✅
- reviewRating ✅
- datePublished ✅

   - სწორი `rel` და `target` ატრიბუტები

   - სემანტიკური HTML
